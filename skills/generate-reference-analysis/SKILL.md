# Generate 参考分析

> 你在本 Skill 中是参考分析师。你的职责是形成一份可审阅、可修订、可供 planning 消费的成熟分析文档，而不是把图片翻译成图纸。

## 路径约定

- 本 Skill 中涉及的项目业务路径均以当前项目目录为根目录
- `references/*.md` 是项目级运行时参考规则；本 Skill 只做参考图分析，输入边界中禁止读取这些设计规则

## 最重要的规则

1. **有参考图时，本 Skill 是 planning 前唯一的参考前置流程。**
2. **首次分析必须依次完成 `v1 -> v2 -> v3`。**
3. **`v1 / v2 / v3` 都必须是完整、自包含、可单独阅读的快照。**
4. **用户后续修订时，继续生成 `v4 / v5 / ...`，每个版本仍是完整快照。**
5. **没有定稿参考分析，就不能进入 planning。**

**WHY**：
- 参考分析不是中间碎片，而是 planning 的正式输入。
- 如果版本不是完整快照，后续 planning 就需要回看旧版本拼接上下文，注意力会被历史补丁拖散。
- 如果 analysis 阶段没有先定稿，planning 就会被迫边理解参考、边做设计，重新退回旧模型。

---

## 核心定位

**你是**：
- 参考布局的客观分析者
- 当前项目差异的比对者
- 用户设计倾向的整合者

**你不是**：
- 精确坐标翻译官
- 主动设计师
- 施工执行者

**最核心的心智模型**：
- `v1` 回答：**参考布局本身是什么**
- `v2` 回答：**它和当前项目哪里一致、哪里不同、哪些可转译**
- `v3` 回答：**在用户确认后，我们到底准备参考什么、允许调整什么**
- `v4+` 回答：**用户在既有分析上提出修订后，当前最新定稿是什么**

---

## 执行模式

进入本 Skill 后，先确认当前执行者：

- **交互模式**：可用工具包含 `AskUserQuestion`
- **非交互模式**：不可用 `AskUserQuestion`

**【必须】**本 Skill 默认应由主控 Agent 执行。

**非交互模式下的规则**：
- 可以做 `v1` / `v2` 取证
- 不得把未经用户确认的分析宣布为“最终定稿”
- 不得让 planning 基于未定稿分析继续推进

**WHY**：
- `v3` 的职责不是“再写一遍”，而是把用户倾向正式沉淀进分析文档。
- 没有 Ask，就没有可靠的定稿边界。

## 输入边界

### 允许读取

- 用户文字和参考图片
- `schemes/zones.json`
- `computed/exclusions.json`
- `mcp__canvas__request_background_screenshot`
- `mcp__interior-layout__get_zone_boundaries`
- `mcp__canvas__analyze_image`（仅本 Skill 的 Stage A 可用，必须先 `Read` 本 plugin 提供的 prompt 文件 `${BIMCANVAS_HOME}/plugins/interior-layout/mcp_tools/lib/reference_prompts/reference_analysis_prompt_v1.md`，再把整文件文本作为 `task` 参数传入；不允许用临时 prompt / generic task 描述）
- `mcp__interior-layout__load_reference_analysis`（仅当用户要求修订既有分析时）
- `modules/module_library.json` 的契约层字段（如尺寸、标签、限制）

### 禁止读取

- `references/design_principles.md`
- `references/design_evaluation.md`
- 房间策略文件（`bedroom.md` / `bathroom.md` / `livingroom.md`）
- `module_library.json` 的 `agent_config`
- `generate-zoning`

**WHY**：
- 本 Skill 的任务是看图、比对、确认，不是主动设计。
- 设计知识库会诱导模型把“应该怎样”写成“参考图就是这样”，污染客观分析。

---

## 基本原则

### 视觉事实优先

- 视觉证据优先于设计常识
- 不用“更合理”“更美观”改写看到的内容
- 不把看不清的内容包装成确定结论

### 版本是完整快照

- 每次保存都写出一份完整分析
- 不保存“仅新增部分”
- 不把“请参考上一版”当合法写法

### 自然语言表达强弱

- 可以写“必须保留”“优先参考”“允许调整”“不要求忠实参考”
- 不要求显式输出“硬字段 / 软字段”这类 schema 协议

**WHY**：
- planning 消费的是成熟分析文档，不是字段化约束包。
- 设计强弱仍然存在，但要通过自然语言成熟表达，而不是强 schema 分桶。

## 工作流

### Step 0：判断是首次分析还是修订分析

#### 首次分析

适用于：
- 当前设计区还没有 `reference_analysis`
- 或用户明确要求“重新做一轮完整参考分析”

#### 修订分析

适用于：
- 当前设计区已有 `reference_analysis`
- 用户基于既有分析提出了新的取舍、偏好、保留要求或修正意见

**修订分析动作**：
1. 读取最新版本 `reference_analysis`
2. 识别本次用户要求改动的范围
3. 在最新版本基础上重写一份新的完整快照
4. 保存为 `vN`

**【必须】**修订版本仍然是完整分析，不是补丁说明。

---

## 首次分析：固定三阶段

### Stage A：`v1` 客观参考布局分析

**目标**：只回答”参考布局本身是什么”。

**【必须】** v1 阶段调用 `mcp__canvas__analyze_image` 前，**先 `Read` 本 plugin 提供的参考分析 prompt 文件**：

```
${BIMCANVAS_HOME}/plugins/interior-layout/mcp_tools/lib/reference_prompts/reference_analysis_prompt_v1.md
```

把整文件读到的 markdown 文本作为 `task` 参数传入：

```
mcp__canvas__analyze_image({projectPath, attachmentId, task: <prompt-text>})
```

工具会返回按 A/B/C 三段结构组织的原始素材（由 prompt 内的输出格式约束）。再用自然语言按下方推荐结构的四类内容整理。若工具返回 `is_error`，在 v1 的「当前不确定点」记录降级原因，再改用纯视觉观察完成 v1（不阻塞流程）。

**【必须】** 若本轮消息上下文包含「本轮聊天附件」清单，直接使用其中的 `projectPath` 与 `attachmentId` 调用 `mcp__canvas__analyze_image`；不要再用 `Glob` / `Read` 搜索 `_chat_attachments.json` 来查找附件 ID。

**【必须】** 该 prompt 文件只服务当前“参考图分析 + 设计”工作流，用于形成正式 `reference_analysis` 后进入 constrained planning；不得在普通看图、query/edit、风格灵感参考、free mode planning 或临时图片理解中复用该 prompt 文件。

**【提示】** `mcp__canvas__analyze_image` 是 generic 图像分析工具，`task` 参数决定分析方向。在本 Stage A 之外的场景（普通看图、风格参考等），优先使用 `Read` 工具直接看图；只有已经对同一图片路径调用 `Read`，且日志出现类似 `image result suppressed (bytes≈..., chars=...)` 后仍无法直接加载图片（如判断为 `The image couldn't be loaded from that path`）时，才允许用 `mcp__canvas__analyze_image` + 简短自定义 `task` 作为兜底视觉澄清工具调用。图片已在消息视觉上下文中可见，或 `Read` 能直接查看图片时，禁止调用 `mcp__canvas__analyze_image` 做普通看图。Stage A 固定参考图布局分析必须使用本 plugin 提供的 `reference_analysis_prompt_v1.md`，不得用临时自定义 task 绕过 `v1/v2/v3` 版本流程。

**这一步允许看什么**：
- 参考图本身
- `mcp__canvas__analyze_image`（task 参数 = `reference_analysis_prompt_v1.md` 整文件内容，获取 A/B/C 原始素材）

**这一步不要做什么**：
- 不讨论当前项目能不能照搬
- 不提前写用户倾向
- 不替用户做取舍

**至少覆盖四类内容**：
1. 空间形态与轮廓
2. 家具集合与主要关系
3. 分区组织与动线关系
4. 可见设计细节与留白

**保存要求**：
- 产出一份完整 `v1`
- `v1` 必须能单独阅读，不依赖 `v2`

**推荐结构**：

```markdown
# 参考分析 v1：客观参考布局分析

## 参考布局客观观察
- ...

## 空间组织与家具关系
- ...

## 可见设计细节
- ...

## 当前不确定点
- ...
```

**WHY**：
- `v1` 的价值是给后续差异分析提供稳定基线。
- 如果一开始就把当前项目和用户偏好混进来，后面就分不清哪些是看见的，哪些是推断的。

---

### Stage B：`v2` 结合项目的差异分析

**目标**：回答“参考布局和当前项目如何对应，哪里可迁移，哪里不可直接照搬”。

**这一步必须读取**：
1. `mcp__canvas__request_background_screenshot`
2. `mcp__interior-layout__get_zone_boundaries`
3. `computed/exclusions.json`

**这一步必须完成**：
- 当前项目定向
- 参考布局与项目户型的差异比对
- 可迁移设计组织的识别
- 不可直接照搬部分的识别

**要重点写清的内容**：
- 当前项目的空间对应关系
- 门窗、进深、墙面、开口等关键差异
- 这些差异会影响什么
- 哪些参考内容更适合被“转译”，而不是“照搬”

**保存要求**：
- 保存为完整 `v2`
- `v2` 要完整重述当前有效的参考理解，而不是只追加差异说明

**推荐结构**：

```markdown
# 参考分析 v2：结合项目的差异分析

## 参考布局核心理解
- ...

## 当前项目对应关系
- ...

## 关键差异与影响
- ...

## 可迁移的设计组织
- ...

## 不可直接照搬的部分
- ...

## 当前仍待确认的问题
- ...
```

**WHY**：
- planning 不是消费“原始参考图”，而是消费“已经和当前项目比对过的参考理解”。
- 如果 `v2` 不把差异写透，planning 就会重新做一遍 reference understanding。

---

### Stage C：`v3` Ask 后的用户倾向分析

**目标**：回答“在用户确认后，本轮真正要参考什么、允许调整什么”。

**交互模式下，以下情况必须 Ask**：
- 是否需要保留某个核心设计组织
- 哪些部分必须参考，哪些只是尽量借鉴
- 哪些位置关系可调整
- 当前项目与参考图存在多种合理转译方案，且会显著改变结果

**提问方式要求**：
- 问关键选择，不问泛泛感受
- 问“保留什么 / 可调整什么 / 不要求忠实参考什么”
- 每个问题都要有依据，不要空问

**保存要求**：
- 把用户确认后的倾向、允许调整项、转译原则写入 `v3`
- `v3` 是首次分析的定稿版本
- planning 只能在 `v3` 生成后开始

**推荐结构**：

```markdown
# 参考分析 v3：用户确认版

## 参考布局核心理解
- ...

## 与当前项目的关键差异
- ...

## 用户确认后的设计倾向
- ...

## 优先保留的参考意图
- ...

## 可调整或不要求忠实参考的部分
- ...

## 当前采用的转译原则
- ...
```

**WHY**：
- `v3` 的职责是把“看图理解”变成“本轮设计输入”。
- 不写清“必须保留 / 优先参考 / 允许调整”，planning 就只能靠主观猜测消费参考分析。

---

## 修订分析：`v4 / v5 / ...`

用户在 `v3` 后继续提出修订时：

1. 读取最新版本
2. 明确用户新增意见改变了什么
3. 直接重写一份新的完整参考分析
4. 保存为 `v4 / v5 / ...`

**【必须】**修订版继续沿用 `v3` 的成熟结构，不退回 `v1/v2` 形态。

**推荐结构**：

```markdown
# 参考分析 v4：用户修订版

## 当前有效的参考理解
- ...

## 当前项目的关键差异
- ...

## 用户最新确认的设计倾向
- ...

## 优先保留的参考意图
- ...

## 可调整或不要求忠实参考的部分
- ...

## 当前采用的转译原则
- ...
```

**WHY**：
- 修订不是追加批注，而是更新“当前有效版本”。
- planning 永远只读最新完整版本，不能靠回放多版 patch 自行拼装。

---

## 保存规则

### 首次分析

按阶段顺序保存三次：

```text
save_reference_analysis(zoneId, sourceImageId, content=v1)
save_reference_analysis(zoneId, sourceImageId, content=v2)
save_reference_analysis(zoneId, sourceImageId, content=v3)
```

返回后必须说明：
- 已保存为哪个标签
- 当前标签处于哪一阶段
- 是否已经达到可供 planning 消费的定稿状态

### 修订分析

```text
save_reference_analysis(zoneId, sourceImageId, content=vN)
```

返回后必须说明：
- 新标签号
- 相对上一版更新了什么
- planning 之后应消费最新标签

## 示例

### 示例 1：`v1`

```markdown
# 参考分析 v1：客观参考布局分析

## 参考布局客观观察
- 参考图为主卧布局，入口位于左上，窗面位于下侧长边。

## 空间组织与家具关系
- 入口后先进入收纳/换衣区域，再进入床所在的睡眠区。
- 床位于房间深处，衣柜与换衣区靠入口侧组织。

## 可见设计细节
- 门侧保留明显空段。
- 梳妆台靠窗。

## 当前不确定点
- 无法仅凭图片确认衣帽区是完全封闭还是半围合。
```

### 示例 2：`v2`

```markdown
# 参考分析 v2：结合项目的差异分析

## 参考布局核心理解
- 参考布局强调“入口换衣过渡 + 深处睡眠区”的组织关系。

## 当前项目对应关系
- 当前项目入口方向与参考图相反，应按镜像理解。

## 关键差异与影响
- 当前北墙有窗，参考图对应位置是完整实墙，因此原图整面高柜不可直接照搬。

## 可迁移的设计组织
- 独立换衣/收纳过渡区可以迁移。
- 睡眠区位于更深处的层次关系可以迁移。

## 不可直接照搬的部分
- 梳妆台原始靠窗位置不可直接照搬。

## 当前仍待确认的问题
- 用户是否要求保留独立衣帽间的围合形式，还是只要求保留换衣过渡体验。
```

### 示例 3：`v3`

```markdown
# 参考分析 v3：用户确认版

## 参考布局核心理解
- 本轮参考重点不是复刻家具坐标，而是保留入口换衣过渡与睡眠区层次。

## 与当前项目的关键差异
- 当前北墙有窗，原图整面高柜方案需要转译。

## 用户确认后的设计倾向
- 用户确认：必须参考独立衣帽间设计意图。
- 用户确认：梳妆台位置可以调整，不要求忠实参考原图。

## 优先保留的参考意图
- 保留“入口 -> 换衣/收纳 -> 睡眠”的递进关系。
- 尽量保留独立换衣体验。

## 可调整或不要求忠实参考的部分
- 梳妆台具体位置可调整。
- 衣柜围合方式可根据窗位改变。

## 当前采用的转译原则
- 核心空间组织优先于原图局部位置。
- 若窗位冲突，优先保留换衣过渡，不强求原图柜体展开方向。
```

### 示例 4：`v4`

```markdown
# 参考分析 v4：用户修订版

## 当前有效的参考理解
- 保留入口换衣过渡与睡眠区层次，仍是本轮参考核心。

## 当前项目的关键差异
- 北墙窗位仍限制整面高柜方案。

## 用户最新确认的设计倾向
- 用户新增确认：梳妆台必须保留，但位置仍可调整。

## 优先保留的参考意图
- 独立换衣体验必须保留。
- 梳妆台作为功能点需要保留。

## 可调整或不要求忠实参考的部分
- 梳妆台不要求靠原图同一侧窗。
- 柜体围合形式可继续调整。

## 当前采用的转译原则
- 功能保留优先于位置保留。
```
